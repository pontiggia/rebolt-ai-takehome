Here it is the full raw workflow I did and all: first we started with claude doing the spec.md. On the spec.md, I detailed everything like the technologies version, the dependencies, using the skills of Vercel, and then I decided to use WorkOS AuthKit for the auth, AI SDK for calling OpenAI provider, I chose the AI SDK with the OpenAI provider because, well, the OpenAI was required, but the AI SDK not. I chose it because it is most simpler than the raw OpenAI API. It has a lot of built-in features that allow me to focus on the actual architecture more than the handling types or whatever.. I also implemented Postgres as it was a requirement, but I choose PlanetScale as provider for Postgres because I used Supabase in the past and other providers, and PlanetScale it was the most efficient of all. I also choose Drizzle as ORM for the database. For the sandbox to render the TypeScript code on the artifact, I used Sandpack, but I regret about that because later on I realized that Sandpack uses some old versions of things and I cannot run like full micro VMs. So something that I will improve is the artifact Sandpack generation. I will use micro VMs. or web containers for the code rendering, or I will also use the Sandpack SDK that I know is paid. From the first moment, I prioritized solid principles, clean code, and the best practices to do it. In terms of the code, and for the database, I created four tables because it's a simple app. One table conversations, one files, another for the messages, and another for the users. Each one has their own indices and also their foreign keys references. For the OpenAI models, I first started with GPT-4.1 Mini as the main orchestrator, and then 5.3 codex for the codegen, but I realized that the 4.1 Mini, it was too small, so I changed it to the 4.1 that has one million context and better capabilities, and the 5.3 codex I changed it for the 5.4 Mini, which is very good. It's the latest model, and it's pretty good at coding and for quick title generation we call gpt 5.4nano. Each one of these models runs a specific tool. Right now we have 3 tools (analyze data -> Inspect full dataset structure/profile and produce summary + insights, readDatasetRows -> Read exact rows from the normalized full dataset with safety caps, generateArtifact -> Produce a multi-file React artifact and inject the dataset helper).
And for the storage, I used and picked Vercel Blob. I knew it from the first moment because it was super easy to integrate and I will deploy to Vercel, so it was the safest and easy choice. On the first test of the agent, I noticed that it was taking too long to build the artifact, and then I find out that the bottleneck was the dataset. The dataset, it was too long, or it could be too long that the agent needs to write all the JSON on the TypeScript code. So I decided that we could parse all the CSV and convert it to a dataset and uploading to the blob. So the TypeScript code only would reference that blob and get all the data. So it could map it or iterate on it or whatever. So the Full rows live in a normalized dataset blob, the model sees only the summary plus tools, and the artifact loads the real dataset at runtime through a helper.
That makes the system smaller, cheaper, more reliable, and more honest.
